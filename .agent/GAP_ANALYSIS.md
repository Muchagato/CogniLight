## Critical Issues

Issues that will cause runtime failures, data loss, or security vulnerabilities.

### [C-001] MarkdownPipe bypasses Angular XSS sanitization on LLM-generated content
- **Layer**: Frontend
- **Files**: `frontend/src/app/shared/pipes/markdown.pipe.ts:16`
- **Symptom**: `bypassSecurityTrustHtml()` is called on HTML generated by `marked.parse()` from LLM responses. An attacker who controls or influences the LLM output (prompt injection) can inject `<script>` tags, `<img onerror>`, or other XSS payloads that execute in the user's browser.
- **Root Cause**: `marked` does not sanitize HTML by default. `bypassSecurityTrustHtml` explicitly tells Angular to skip its DomSanitizer.
- **Impact**: XSS vulnerability — stolen API keys from localStorage, session hijacking, arbitrary actions in the user's browser context. The BYOK API key stored in localStorage is the highest-value target.
- **Evidence**: `return this.sanitizer.bypassSecurityTrustHtml(html);` on line 16.

### [C-002] IncidentLogGenerator.Queue accessed from multiple timer threads without synchronization
- **Layer**: Backend
- **Files**: `backend/CogniLight.Api/Services/IncidentLogGenerator.cs:21,79-85,96-98`
- **Symptom**: `_pendingAnomalies` (a `Queue<T>`) is written to in `OnAnomalyDetected()` (called from SimulationEngine's timer thread) and read/dequeued in `Tick()` (called from IncidentLogGenerator's own timer thread). `Queue<T>` is not thread-safe.
- **Root Cause**: Two `System.Threading.Timer` callbacks run on thread pool threads without any locking around the shared queue.
- **Impact**: Potential `InvalidOperationException`, corrupt queue state, or lost anomaly events under concurrent access. The 30-second tick interval makes collisions infrequent but possible — especially during anomaly bursts.
- **Evidence**: `private readonly Queue<...> _pendingAnomalies = new();` — no `lock`, no `ConcurrentQueue`.

### [C-003] SQLite not configured for WAL mode — concurrent reads may block
- **Layer**: Cross-cutting (Backend + AI Service)
- **Files**: `backend/CogniLight.Api/Program.cs:24-27`, `ai-service/main.py:48-52`
- **Symptom**: The architecture docs state "WAL mode handles one writer and multiple readers safely", but no code sets `PRAGMA journal_mode=WAL`. EF Core SQLite defaults to DELETE journal mode.
- **Root Cause**: WAL mode must be explicitly enabled. Neither the backend connection string nor the AI service SQLAlchemy engine configures it.
- **Impact**: In DELETE mode, readers (AI service) are blocked when the writer (backend) holds a transaction lock during the 1-second save cycle. This causes intermittent timeouts on AI service SQL queries, especially under load.
- **Evidence**: No `journal_mode` in connection string `"Data Source=cognilight.db"` or in any `PRAGMA` call.

## Major Issues

Issues that degrade quality, maintainability, or developer experience significantly.

### [M-001] Frontend REST methods don't check response status before parsing JSON
- **Layer**: Frontend
- **Files**: `frontend/src/app/shared/services/telemetry.service.ts:184-212`
- **Symptom**: `getHistory()`, `getPoleHistory()`, `getAnomaliesInRange()`, `getIncidentLogs()`, `getIncidentLogsInRange()` all call `fetch()` and immediately call `resp.json()` without checking `resp.ok`. A 429 (rate limit), 500 (server error), or network error will try to parse the error body as JSON, likely throwing a confusing parse error instead of a meaningful message.
- **Root Cause**: Missing `resp.ok` checks after fetch.
- **Impact**: Dashboard silently fails with cryptic errors when backend returns non-200 responses. User sees blank charts with no feedback.
- **Evidence**: `const resp = await fetch(...); return resp.json();` — no status check.

### [M-002] CORS inconsistency between backend and AI service defaults
- **Layer**: Cross-cutting
- **Files**: `backend/CogniLight.Api/Program.cs:38-41`, `ai-service/main.py:38`
- **Symptom**: When `CORS_ORIGINS` env var is not set: backend defaults to `["http://localhost:4200"]`, AI service defaults to `["*"]` (allow all origins).
- **Root Cause**: Different fallback logic in each service.
- **Impact**: In development, the AI service is open to all origins while the backend is restricted. In production, if `CORS_ORIGINS` is misconfigured or empty, the AI service becomes an open endpoint accepting requests from any domain. This matters because the AI service receives the user's LLM API key in headers.
- **Evidence**: Backend: `new[] { "http://localhost:4200" }`. AI: `["*"]`.

### [M-003] LLM max_tokens capped at 500 — truncates detailed analysis responses
- **Layer**: AI Service
- **Files**: `ai-service/rag/chain.py:242,258,286,320`
- **Symptom**: All LLM calls (both streaming and non-streaming, both Anthropic and OpenAI) use `"max_tokens": 500`. The system prompt asks for "tables when appropriate" and "data-driven" analysis, but a table of 12 poles with multiple columns can easily exceed 500 tokens.
- **Root Cause**: Hardcoded limit across all 4 call sites.
- **Impact**: Responses are frequently truncated mid-sentence or mid-table. Users see incomplete analysis with no indication it was cut off.
- **Evidence**: `"max_tokens": 500` in `_call_anthropic`, `_call_openai`, `_stream_anthropic`, `_stream_openai`.

### [M-004] GetLatestReadingsAsync uses N+1 query pattern (12 individual queries)
- **Layer**: Backend
- **Files**: `backend/CogniLight.Api/Services/TelemetryService.cs:35-54`
- **Symptom**: Iterates through 12 pole IDs and executes a separate `FirstOrDefaultAsync` query for each. This creates 12 round-trips to SQLite per call.
- **Root Cause**: LINQ approach with per-pole loop instead of a single GROUP BY query.
- **Impact**: Unnecessary overhead on every `/api/telemetry/latest` call. Each call makes 12 DB queries instead of 1. Not critical for SQLite (local file) but bad practice and slower than necessary.
- **Evidence**: `foreach (var poleId in poleIds) { var latest = await db.TelemetryReadings.Where(r => r.PoleId == poleId).OrderByDescending(...).FirstOrDefaultAsync(); }`

### [M-005] Missing backend `/health` endpoint
- **Layer**: Backend
- **Files**: `backend/CogniLight.Api/Program.cs`
- **Symptom**: No health check endpoint exists. The AI service has `/health` but the backend doesn't.
- **Root Cause**: Never implemented.
- **Impact**: No way to monitor backend availability. Docker health checks can't be configured. Reverse proxies and load balancers have no health probe target.
- **Evidence**: Grep for `/health` in backend returns no results.

## Improvement Opportunities

### [I-001] Add Docker health checks to compose files
- **Layer**: Infrastructure
- **Current State**: Neither `docker-compose.yml` nor `docker-compose.prod.yml` define health checks.
